# Pipeline Builder runbook — zero-thinking edition

Two pipelines. Pipeline 2 is the one that matters for the demo: it is fully
deterministic and produces the recommendations. Pipeline 1 is display-only
enrichment. Build Pipeline 2 first.

Source files referenced (all in `foundry_export/`):
- `pipeline_protocol_gap_transform.py` — paste verbatim into the Pipeline 2 transform node.
- `02_pipeline_and_function_spec.md` — the authoritative spec.

## Confirmed entry point (Pipeline 2)

```python
def transform(notes: "DataFrame") -> "DataFrame"
```

- **Entry-point function name to set in the transform node:** `transform`
- **Input:** a single argument `notes`, a Spark **DataFrame**. The transform
  only reads two columns and ignores the rest:
  - `patient_id` (string)
  - `note_text` (string)
  It does `notes.select("patient_id", "note_text").rdd.mapPartitions(...)`,
  so the input dataset MUST expose columns literally named `patient_id` and
  `note_text`. It is NOT a list and NOT a pandas DataFrame — it is a pyspark
  DataFrame.
- **`protocols` is NOT a runtime input.** The protocol library and all
  negation/context rules are inlined in the file. Do not wire a `protocols`
  source into this node. (The spec lists `protocols` for traceability only;
  the file's docstring confirms: "protocols: not used at runtime.")
- **Output:** a Spark DataFrame matching `OUTPUT_SCHEMA`, one row per
  (patient, triggered protocol, missing step). Columns, in order:

  | column            | type    | notes |
  |-------------------|---------|-------|
  | patient_id        | string  |       |
  | protocol_key      | string  | sepsis, acs, stroke, cap, dka, pe, gi_bleed, aki, ciwa, neutropenic_fever, hyperkalemia, copd |
  | protocol_name     | string  |       |
  | action_key        | string  |       |
  | action_label      | string  | e.g. "Administer broad-spectrum antibiotics" |
  | action_severity   | string  | required \| recommended |
  | urgency           | string  | red \| amber \| green |
  | owner             | string  | physician \| pharmacist \| nurse \| case_manager |
  | citation          | string  | e.g. "Surviving Sepsis Campaign Hour-1 Bundle (2018)" |
  | trigger_pattern   | string  | the regex that fired |
  | trigger_evidence  | string  | literal substring of the note |
  | trigger_start     | integer | char offset in note_text |
  | trigger_end       | integer | char offset in note_text |

  If a protocol triggers but every expected step is documented, it emits no
  rows for that (patient, protocol). If nothing triggers, the patient simply
  has no rows.

## Pipeline 2 — protocol gap detection

> **READ THIS FIRST — which product.** The file
> `pipeline_protocol_gap_transform.py` is written in the **Code Repositories /
> Python Transforms** style: a top-level `def transform(notes) -> DataFrame`
> with module-level `from pyspark.sql import ...`, its own
> `SparkSession.builder.getOrCreate()`, and a return of
> `spark.createDataFrame(rdd, schema=OUTPUT_SCHEMA)`. **Pipeline Builder's
> "Use Python" / Python-transform node does NOT accept a self-contained
> module like this** — that node surface is for column-level pyspark
> expressions / UDFs with a fixed injected signature, not a pasted file that
> creates its own SparkSession and returns a DataFrame from an RDD. If you
> open Pipeline Builder expecting a "paste a .py file and name an entry point"
> box, you will not find one and you will stall on camera.
>
> **Build this transform in a Python Transforms repository (Code
> Repositories), not in Pipeline Builder.** The Code Repositories template for
> a Python transform already has the `@transform` / `@transform_df`
> decorator and a function that receives/returns datasets — paste the body
> there. The two approaches below both work; pick A.

### Path A (recommended) — Python Transforms repository

1. **New repo.** Create a Code Repository → Python Transforms template.
2. **Add the code.** Replace the template's transform module with the WHOLE
   `pipeline_protocol_gap_transform.py` verbatim (pyspark + stdlib only;
   never hand-edit — it is generated by `sync_transform.py` and parity is
   enforced in CI). Add the input/output binding the template expects, e.g.:

   ```python
   from transforms.api import transform, Input, Output

   @transform(
       gaps=Output("/<your-project>/protocol_gaps"),
       notes=Input("/<your-project>/notes"),
   )
   def compute(gaps, notes):
       gaps.write_dataframe(transform(notes.dataframe()))
   ```
   (Rename the wrapper if it would shadow the imported `transform`; e.g.
   `from pipeline_protocol_gap_transform import transform as run_gaps`.)
   The input dataset MUST expose columns literally named `patient_id` and
   `note_text` — `notes.select("patient_id","note_text")` will fail loudly
   otherwise. Rename upstream if your dataset differs.
3. **Output dataset.** `protocol_gaps`. Schema is fixed by `OUTPUT_SCHEMA`
   (13 columns above) and set explicitly in `createDataFrame` — do not
   re-declare it by hand.
4. **Build.** Run a build (or preview). Expect zero-to-many rows per patient
   across the 176-note corpus. Sanity check: a sepsis note missing the
   antibiotics step yields `protocol_key=sepsis`, `urgency=red`, populated
   `trigger_evidence` and integer offsets.
5. **Publish/commit** so `protocol_gaps` materializes for downstream objects
   and the `classify_bottleneck` function.

### Path B — Pipeline Builder (only if you must stay in PB)

You cannot paste the file. You would have to re-express the logic as a
Pipeline Builder Python UDF (per-row, returning an array of structs) and
then explode it — a non-trivial rewrite that diverges from the CI-frozen
file and breaks parity. **Do not do this for the demo.** Use Path A.

Either way: do NOT wire `protocols` as an input — the transform inlines the
rules. (See the CSV import note below before trusting `notes`.)

### If you ever do need protocols as input
You don't, for this transform. The rules are baked in. If a future variant
must read protocols at runtime, you would add a second input and change the
`transform` signature to accept it — but that diverges from the
CI-frozen generated file, so don't. Re-run `sync_transform.py` instead.

## CSV import note — multi-line quoted `notes.csv`

`note_text` cells contain newlines (multi-line clinical notes). When you
import `notes.csv` as a dataset:

- Use a CSV parser with **proper RFC-4180 quoted-field parsing** (quoted
  cells may span multiple physical lines).
- In the Foundry CSV import settings, enable "multiline" / quoted-field
  handling and set the quote character to `"`. Do NOT use a naive
  newline-splits-rows parser.
- Verify after import: row count must equal **176** (the corpus has exactly
  176 notes / 176 unique `patient_id`, one row per note — confirmed by
  parsing `notes.csv` with a real CSV reader). The raw file is ~905 physical
  lines because of embedded newlines; if your imported dataset shows ~905
  rows, or anything other than 176, the parser split multi-line cells and you
  must re-import with multiline/quoted-field handling on. Also confirm
  `note_text` for a known multi-paragraph note (e.g. `P-1105`) is intact in a
  single cell.

A broken import here silently shreds notes into fragments, which changes
trigger offsets and breaks parity with the local engine.

## Pipeline 1 — note enrichment (LLM node vs deterministic Python)

Inputs: `notes`, `icd10_reference`. Output: `note_features` (one row per
note). This is **display-only enrichment** (vitals/labs JSON, ICD-10
candidates) — it does NOT feed the recommendation path.

Two ways to build each step:

- **LLM node (spec default):** an LLM node for entity extraction (system
  prompt in the spec, user prompt `{note_text}`), and a second LLM node for
  ICD-10 candidate ranking against `icd10_reference`. Fine here because the
  output is only shown, never turned into a recommendation.
- **Deterministic Python transform (recommended for the story):** port
  `app/nlp/extractor.py` as a Python transform node for extraction, and a
  TF-IDF Python transform (the local scikit-learn version) for ICD-10
  candidates. This keeps the enrichment layer LLM-free end to end.

**Recommendation:** use the deterministic Python path so you can truthfully
say **"no LLM is in the path that produces a recommendation"** — and, with
the deterministic enrichment, no LLM anywhere. If you prefer the LLM nodes
for richer extraction in the demo UI, that is acceptable *only* for this
display-only enrichment; the recommendation path (Pipeline 2 +
`classify_bottleneck`) must stay deterministic regardless.

`note_features` columns: `patient_id, vitals_json, labs_json, consult_*,
imaging_pending, placement_need, isolation_required, explicit_diagnoses,
icd10_top5_json`.
